# GovernanceIQ Education Data & Content Governance API — Technical Design

## 1) Scope and locked requirements

This backend centralizes **institutional analytics** and **data/content governance operations** in one offline-deployable service using Django REST Framework + MySQL.

Locked requirements reflected in this design:

- Hierarchical RBAC: **Administrator ⊃ Operations ⊃ User**.
- Audit-log export remains **Administrator-only**.
- “Query approved datasets” is **governed API querying/filtering** (no arbitrary raw SQL execution).
- Export jobs split output into multiple parts when total rows exceed **250,000 rows/file**.
- Monitoring computes recommendation CTR from **local impression/click event logs only**.
- Alumni-facing content includes **poetry and tribute entries**.
- Ticket state machine is exactly: `open → in_progress → blocked/resolved → closed` (with guarded transitions).
- Metadata fields are explicit: `owner`, `retention_class`, `sensitivity_level`.
- Datasets/fields have stable internal IDs; lineage edges include timestamps; content version history stores operator, timestamp, changed fields, and reason.
- Inspection schedule default is nightly at **02:00 local time**.

## 2) Runtime and deployment contract

- Primary runtime: `docker compose up --build`
- Database initialization: `./init_db.sh`
- Broad test command: `./run_tests.sh`
- No checked-in `.env*` files. Runtime secrets/config are injected via Docker secrets/mounted files generated by a local bootstrap path, not committed to source.

### 2.1 Offline deployment topology (single host)

- `api`: Django + DRF (Gunicorn)
- `worker`: Celery worker
- `scheduler`: cron-driven scheduler container that calls
  `apps.platform_common.scheduler.tick_all` once per minute
  (configurable via `--interval`). On each tick it fires every active
  `InspectionSchedule` and `ReportSchedule` whose `next_run_at <= now`,
  produces real `InspectionRun`/`ReportRun` rows, and advances
  `next_run_at` (computed via `croniter` from the schedule's
  `cron_expr` + `timezone`) plus `last_enqueued_at` inside the same DB
  transaction. The same code path is exposed manually via
  `python manage.py run_scheduler_tick`.
- `mysql`: MySQL 8.x
- `redis`: local broker/cache
- `proxy`: Nginx (TLS termination on 443; HSTS; HTTP→HTTPS redirect; TLS 1.2/1.3 only)
  - cert/key generated by `bootstrap_runtime` for local-dev only; production
    deployments must replace with platform-managed certificates
- attached local storage mounts:
  - `/var/lib/governanceiq/backups`
  - `/var/lib/governanceiq/exports`

## 3) Architecture and module boundaries

Modular monolith (single Django project with domain apps):

1. `identity`: users, sessions/tokens, login/logout/revoke
2. `authorization`: role inheritance and capability enforcement
3. `catalog`: datasets, fields, metadata
4. `lineage`: lineage edges/graph traversal
5. `quality`: rules, inspections, scoring, schedules
6. `tickets`: issue lifecycle, remediation actions, backfills
7. `content`: alumni content entries (`poetry` + `tribute`) and versions
8. `analytics`: governed dataset querying + report definitions/runs
9. `exports`: CSV/XLSX artifact jobs with retention/splitting
10. `audit_monitoring`: append-only audit log + local metrics/CTR
11. `platform_common`: idempotency, optimistic concurrency, validation, encryption helpers, shared errors

### 3.1 Cross-cutting policies

- **Idempotency**: required on create/update operations that may duplicate tickets, backfills, report runs (24-hour dedupe window).
- **Optimistic concurrency**: mutable records carry `version`; writes require `If-Match` with expected version.
- **Immutable evidence**: inspection results and audit logs are append-only.

## 4) Role and capability model

Roles are hierarchical unless explicitly restricted:

- `administrator`: all `operations` + all `user` + restricted admin capabilities
- `operations`: all `user` + operational governance capabilities
- `user`: approved dataset querying/report execution + published content read

### 4.1 Capability matrix (summary)

- **Administrator**
  - user/role management, permission grants
  - encryption/backup settings
  - audit-log export (**admin-only restriction**)
  - all operations/user capabilities
- **Operations**
  - onboarding datasets/fields/metadata
  - quality rules and inspections
  - ticket triage/remediation/backfill
  - report scheduling and export operations
  - content draft/publish/rollback
  - all user capabilities
- **User**
  - query/filter approved datasets via governed API
  - run permitted reports
  - read published poetry/tribute entries

## 5) Domain data model and constraints

All major entities use stable internal IDs (`CHAR(26)` ULID or `BINARY(16)` UUID) and server timestamps.

### 5.1 Identity and access

- `users(id, username UNIQUE, password_hash, is_active, created_at, updated_at, version)`
  - password hashing: Argon2id (salted)
- `roles(id, name UNIQUE)` with seeded values: `administrator`, `operations`, `user`
- `user_roles(user_id, role_id, UNIQUE(user_id, role_id))`
- `permission_grants(id, principal_type, principal_id, capability, scope_json, granted_by, created_at)`
- `sessions(id, user_id, token_hash UNIQUE, expires_at, revoked_at NULL, ip, user_agent, created_at)`
  - 8-hour expiry; revocable

### 5.2 Catalog, metadata, lineage

- `datasets(id, code UNIQUE, display_name, approval_state, created_by, created_at, updated_at, version)`
- `dataset_fields(id, dataset_id, field_key, display_name, data_type, is_queryable, created_at, updated_at, version, UNIQUE(dataset_id, field_key))`
- `dataset_metadata(id, dataset_id UNIQUE, owner, retention_class, sensitivity_level, updated_by, updated_at, version)`
- `lineage_edges(id, upstream_dataset_id, downstream_dataset_id, relation_type, observed_at, recorded_at)`
  - indexed on `(upstream_dataset_id, recorded_at)` and `(downstream_dataset_id, recorded_at)`

### 5.3 Quality governance

- `quality_rules(id, dataset_id, rule_type, severity, threshold_value, config_json, active, created_by, created_at, updated_at, version)`
  - `rule_type`: `completeness | consistency | uniqueness | numeric_range | distribution_drift`
  - `severity`: `P0 | P1 | P2 | P3`
- `quality_rule_fields(id, rule_id, dataset_field_id)`
- `inspection_runs(id, dataset_id, trigger_mode, started_at, ended_at, quality_score, gate_pass, status, created_by)`
- `inspection_rule_results(id, inspection_run_id, rule_id, measured_value, threshold_snapshot, severity_snapshot, weight_snapshot, passed, breach_delta, created_at)`
  - immutable after insert
- `inspection_schedules(id, dataset_id, cron_expr, timezone, active, next_run_at, updated_at, version)`
  - default: `0 2 * * *` in local timezone

### 5.4 Tickets and remediation

- `issue_tickets(id, dataset_id, inspection_run_id, rule_result_id, title, description, state, owner_user_id, due_date, severity_snapshot, created_at, updated_at, version)`
  - state machine: `open`, `in_progress`, `blocked`, `resolved`, `closed`
  - default due date: `created_at + 7 calendar days`
- `ticket_transitions(id, ticket_id, from_state, to_state, reason, transitioned_by, transitioned_at)`
- `remediation_actions(id, ticket_id, action_type, parameters_json, status, created_by, created_at, updated_at, version)`
- `backfill_runs(id, ticket_id, input_fingerprint, affected_record_count, started_at, ended_at, status, operator_user_id, created_at)`
  - idempotent by `(ticket_id, input_fingerprint)` within 24h key window
- `backfill_reinspections(id, backfill_run_id, inspection_run_id, created_at)`

### 5.5 Alumni content governance (poetry + tribute)

- `content_entries(id, content_type, slug UNIQUE, title, current_published_version_id NULL, created_by, created_at, updated_at, version)`
  - `content_type`: `poetry | tribute`
- `content_versions(id, entry_id, body, state, operator_user_id, changed_fields_json, reason, created_at)`
  - `state`: `draft | published | rolled_back`
  - publish requires `reason` length >= 10 chars
  - version history stores operator, timestamp, changed fields, reason

**Single published version constraint:**

- Enforced in service transaction + DB unique index using generated key:
  - `published_key = CASE WHEN state='published' THEN CONCAT(entry_id) ELSE NULL END`
  - unique index on `published_key`
  - multiple `NULL` allowed, one published row per entry key

**Rollback window:**

- Rollback allowed only to versions created within last 30 days.

### 5.6 Analytics, reporting, exports

- `report_definitions(id, name, dataset_id, filter_schema_json, time_window_schema_json, permission_scope_json, query_plan_json, created_by, created_at, updated_at, version)`
- `report_runs(id, report_definition_id, requested_by, resolved_filters_json, resolved_time_window_json, status, started_at, ended_at, total_rows, created_at)`
- `report_schedules(id, report_definition_id, cron_expr, timezone, active, next_run_at, last_enqueued_at, created_by, created_at, updated_at, version)`
  - first-class persisted scheduled-run record (default cron `0 3 * * *`)
- `export_jobs(id, report_run_id, format, status, total_rows, file_count, expires_at, requested_by, created_at, completed_at)`
  - row cap: 250,000 rows/file
- `export_files(id, export_job_id, part_number, row_count, path, checksum_sha256, created_at)`
  - multi-part output under one export job when row cap exceeded

### 5.7 Audit and monitoring

- `audit_logs(id, actor_user_id, action, object_type, object_id, request_id, ip, payload_before_json, payload_after_json, created_at)`
  - append-only
- `event_logs(id, event_type, event_ts, actor_user_id NULL, dataset_id NULL, payload_json)`
  - includes recommendation impression/click events
- `metric_rollups(id, period_start, period_end, metric_name, metric_value, dimensions_json, computed_at)`

## 6) Core business logic

### 6.1 Quality scoring and pass/fail gate

- Severity weights (default): `P0=50`, `P1=30`, `P2=15`, `P3=5`.
- Each rule result yields normalized compliance `r` in `[0,1]`.
- Score:

`quality_score = round(100 * sum(weight_i * r_i) / sum(weight_i), 2)`

- Gate rule: if **any P0** rule breaches threshold, `gate_pass = false` regardless of score.

### 6.2 Rule semantics defaults (resolved)

- `completeness`: percent non-null/non-empty ≥ threshold
- `consistency`: declarative predicate checks over configured fields with allowed operators (`=`, `!=`, `<`, `<=`, `>`, `>=`, regex, set-membership, conditional implication); no arbitrary SQL
- `uniqueness`: duplicate ratio ≤ threshold
- `numeric_range`: out-of-range ratio ≤ threshold and optional hard min/max bounds
- `distribution_drift`: Population Stability Index (PSI) ≤ threshold using baseline built from trailing 30 approved runs, refreshed nightly. Rule activation requires at least 10 approved historical runs; before that, rule remains inactive and cannot be attached as enforced.

### 6.3 Ticket transition policy

Allowed transitions:

- `open -> in_progress | blocked | resolved`
- `in_progress -> blocked | resolved`
- `blocked -> in_progress | resolved`
- `resolved -> closed | in_progress`
- `closed` terminal (no reopen transition in v1)

Every transition requires reason + audit entry.

### 6.4 Content publish and rollback flow

Publish transaction:

1. validate reason length >= 10
2. lock entry row (`SELECT ... FOR UPDATE`)
3. set prior published version(s) to non-published state (if present)
4. set target version state `published`
5. update `current_published_version_id`
6. append audit log

Rollback transaction:

1. validate target version age <= 30 days
2. create new version (state `published`) representing rollback result
3. mark replaced published version accordingly
4. append audit log

### 6.5 Governed dataset query guardrails (resolved defaults)

- Per-request row limit: `1..5000` (default 500 when omitted)
- Max filter clauses: `20`
- Max sort fields: `3`
- Max query execution budget: `30s` server-side timeout
- All filter/sort fields must be allowlisted per dataset and permission scope
- Deny unbounded full scans for non-approved datasets or disallowed fields

## 7) Security boundaries

- TLS required for all network transport through proxy.
- Sensitive fields encrypted at rest with AES-256-GCM field-level encryption.
  - key material loaded from local secret mount; key IDs/versioning stored with ciphertext.
  - rotation policy: quarterly rotate active key; decrypt-on-read, re-encrypt-on-write + background rewrap job.
- DRF throttling defaults:
  - authenticated user: 120 req/min
  - IP: 30 req/min
- Strict serializer validation and allowlisted filter grammar prevent SQL injection.
- Content payload sanitization/escaping policy prevents stored XSS in downstream renderers.
- Audit export endpoint restricted to Administrator.

## 8) Reliability, backup, and restore

- Nightly encrypted backups to local attached storage.
  - Implementation: `apps/platform_common/backup.py` shells `mysqldump`
    (with `--single-transaction --quick --routines --triggers
    --set-gtid-purged=OFF`), encrypts with AES-256-GCM under
    `BACKUP_ENCRYPTION_KEY`, and writes `<label>.sql.enc` plus
    `<label>.manifest.json` (algorithm, key id, plaintext + ciphertext
    SHA-256, plaintext bytes, source database, creation timestamp,
    `pitr_retention_days`).
  - Operator path: `python manage.py run_backup` (cron-installable via
    `./scripts/install_backup_cron.sh`).
- MySQL binlog retention supports point-in-time restore (PITR) for last 14 days.
  - Implementation: docker-compose launches MySQL with `--log-bin`,
    `--binlog-format=ROW`, `--binlog-expire-logs-seconds=1209600`, GTID on.
- Restore target: recover to any point within 14-day window.
  - Operator path: `python manage.py run_pitr --target-time <ISO-8601>
    --target-database <db> [--source-database <db>] [--dry-run] [--json]`
    (wrapper `./scripts/pitr_restore.sh`).
  - Implementation: picks the latest backup at-or-before `target_time`
    from the manifest set, restores it into `target_database`, then
    streams binlogs via `mysqlbinlog --read-from-remote-server` filtered
    by `[backup_time, target_time)` and applied into the target. When
    source ≠ target on the same MySQL server, the command uses
    `--rewrite-db source->target` and `--skip-gtids` so events apply as
    fresh transactions in the recovery database.
- Scheduled retention jobs:
  - exports purge after 30 days
  - idempotency keys purge after 24 hours
  - expired sessions cleanup

### 8.1 Backup encryption default (resolved)

- Backups encrypted with AES-256 using rotating backup key in local secret mount.
- Backup manifest stores checksum + key-id metadata.
- Monthly restore drill mandated in operations runbook.

## 9) API consistency constraints (shared)

- Auth: bearer token session, 8-hour expiry.
- Idempotency header: `Idempotency-Key` on duplicate-prone writes.
- Concurrency header: `If-Match` required for mutable resources using `version`.
- Error envelope: `{ "error": { "code", "message", "details", "request_id" } }`.
